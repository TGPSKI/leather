DESCRIPTION

Under p99 load tests, the checkout-api was saturating the Redis connection
pool during peak traffic windows. With the default pool size of 8, concurrent
goroutines were blocking on connection acquisition for up to 340ms, which
cascaded into checkout timeouts and cart abandonment spikes.

Root cause analysis:
  - Each checkout flow makes 3-5 Redis calls (session read, inventory lock,
    cart merge, promo validation, rate-limit check).
  - At peak load, ~250 concurrent checkout goroutines require up to 1250
    Redis connections simultaneously.
  - With pool=8, queue depth hit 242 waiters at 13:47 UTC on 2026-05-12
    (Black Friday simulation, 3x normal traffic).
  - p99 checkout latency spiked from 180ms to 1420ms during the window.

This PR does three things:
  1. Bumps pool_size from 8 to 64.
  2. Adds a health monitor goroutine that logs pool saturation warnings when
     the wait queue exceeds 50% of pool capacity (configurable threshold).
  3. Adds a circuit-breaker middleware that returns HTTP 503 with
     Retry-After: 2 when the queue depth exceeds a hard ceiling of 200.

Load test results (staging, 3x traffic multiplier):
  Before: p50=82ms p95=390ms p99=1420ms  errors=1.3%
  After:  p50=79ms p95=195ms p99=310ms   errors=0.04%

Redis cluster impact:
  pool=64 x 12 app instances = 768 total connections
  Current cluster max_clients=10000
  Peak observed connections before this PR: ~96 (pool=8 x 12 instances)
  After: estimated peak ~768 (7.7% of cluster max_clients)

Connection lifecycle:
  - idle_timeout set to 4 minutes (was 0 / never)
  - max_connection_age set to 30 minutes to prevent stale connections
  - health check interval: 60 seconds (new; pings idle connections)

+func (m *HealthMonitor) Start(ctx context.Context, log *logging.Logger) {
+    go func() {
+        ticker := time.NewTicker(m.interval)
+        defer ticker.Stop()
+        for {
+            select {
+            case <-ctx.Done():
+                return
+            case <-m.stop:
+                return

+                    log.Warn("redis pool saturation warning",
+                        "depth", depth, "ratio", ratio)

+            }
+        }

times 12 instances = 768 connections, which is within cluster limits but I

Also: is there a risk that increasing pool size just shifts the bottleneck
to Redis CPU rather than our wait queue?

@rwilson [2026-05-14 10:28]
Good question. I ran a Redis CPU profile during the 3x load test:
  Before (pool=8):  Redis CPU 12% average, 41% peak (spike from reconnects)
  After (pool=64):  Redis CPU 9% average, 23% peak
The reduced reconnect churn (reconnects drop from ~400/min to ~15/min)
actually lowered overall Redis CPU despite more simultaneous connections.
The key insight is that short-lived connections are more expensive than
persistent idle ones when the cluster is IO-bound.

@mpark [2026-05-14 11:34]
On the WaitQueueDepth() implementation: pool.Stats().Waits returns a
cumulative counter, not a point-in-time gauge. So on the first call after
startup you get 0, but after the first saturation event it grows
monotonically and never resets. The circuit breaker will then be permanently
tripped once you have ever had 200 cumulative waits in the process lifetime,
which could be minutes into a normal deploy.

Is this intentional? If not, you need to track the delta between successive
readings, or use a different stats field.

@rwilson [2026-05-14 11:52]
@mpark - critical bug, thank you. You are absolutely right. pool.Stats()
exposes:
  - Hits:         cumulative connections served from pool (not waited)
  - Misses:       cumulative connections that needed a new connection
  - Timeouts:     cumulative timed-out waits
  - Waits:        cumulative wait events (exactly as you described)
  - WaitDuration: cumulative duration of all waits

None of these are point-in-time gauges. The correct approach is to compute
a sliding delta:

  func (c *Client) WaitQueueDepth() int {
      stats := c.pool.Stats()
      current := stats.Waits
      prev := atomic.SwapInt64(&c.lastWaits, current)
      return int(current - prev)
  }

This gives the number of new wait events since the last poll, which is a
reasonable proxy for current queue depth when sampled frequently.

However, this is still not ideal because it is rate-dependent. If we poll
every 30s, a burst of 300 waits in 1 second will look the same as a slow
accumulation over 30s. For the circuit breaker we probably want a different
signal: maybe pool.Stats().IdleConns == 0 as a simple "pool is empty" check.

Will push a fix for this in the next commit.

@mpark [2026-05-14 12:08]
The delta approach is better but as you note, still bursty. For the hard
ceiling I suggest:
  saturated := stats.TotalConns >= int32(cfg.PoolSize) && stats.IdleConns == 0
This is deterministic and correct: the pool is at capacity AND all connections
are in use. No timing dependency.

@rwilson [2026-05-14 12:31]
Updated. New implementation:

  func (c *Client) IsSaturated() bool {
      stats := c.pool.Stats()
      return stats.TotalConns >= int32(c.cfg.PoolSize) &&
             stats.IdleConns == 0
  }

Circuit breaker now calls IsSaturated() instead of comparing WaitQueueDepth
to a ceiling. Pushed as 3f8a22b.

@kchen [2026-05-14 12:44]
